CSE 397 Professional Career Project - Course Overview and Expectations

This class meeting covers the structure, expectations, and technical workflows for CSE 397, a professional career project course that serves as an internship substitute. The instructor outlines weekly time commitments (9-12 hours), team roles, GitHub version control practices, Agile methodology, and the six-iteration product development cycle students will complete.

Course Structure and Weekly Time Expectations

The course requires 9-12 hours per week of work, structured as: (1) 1 hour class meeting, (2) 1 hour team project meeting, (3) 30 minutes to 1 hour on Gold Certificate career preparation, (4) 30 minutes to 1 hour on career readiness activities (job search, AWS training, ethics), and (5) remaining hours on actual product development. The instructor emphasizes this is a low-cost alternative to $5,000-$10,000 professional training programs, calculated at approximately $35 per week based on tuition costs.123

Team Roles and Responsibilities

Each team has 7 defined roles (with 5 team members, so some roles are shared): Product Owner (customer contact point), Scrum Master (ensures tasking is organized), Front-end Tech Lead, Back-end Tech Lead, Database Tech Lead (coordinate to prevent misalignment), Quality Assurance (verifies requirements are met), and Configuration/Delivery Manager (ensures proper product delivery). All roles are tracked in a role planning report and documented in GitHub issues.4

GitHub Version Control and Branching Strategy

Students use GitHub for version control with a specific branching strategy: each developer has an individual branch, which merges weekly into development week XX branches (e.g., Delivery 0.0 for weeks 1-2). The instructor merges these into the main development branch every odd week starting at week 3. This prevents developers from 'clobbering' each other's code and allows conflict resolution at the individual level before integration. The workflow is: pull current code → merge changes → push new code.567

Agile Story Format and Task Management

All work is tracked as Agile stories stored in GitHub issues using the format: 'As a [stakeholder], I want [behavior], so that [business value].' Each story includes story points (complexity estimation), definition of done (completion criteria), and is organized by the 5 Ws and H (Who, What, Where, When, Why, How). Stories are broken into subtasks and tracked through a Kanban board (To-Do, In Progress, Done) with milestones for each week.8910

Full-Stack Development and CRUD Operations

The course teaches full-stack development with three tiers: Front-end (user interface), Back-end/Server (business logic and APIs), and Database (data storage). The Model-View-Controller (MVC) pattern is applied at each level. Every feature implements CRUD operations: Create (new data), Read (retrieve data), Update (modify data), and Delete (remove data). Teams coordinate across front-end, back-end, and database leads to ensure features are fully supported at all levels.1112

Six-Iteration Product Development Cycle

The semester consists of 6 iterations/versions of the product, each with specific deliverables: Version 1 (Elicitation & Requirements documents), Version 2 (Design document), Versions 3-5 (Test plan, installation guide, user guide, and implementation), and Version 6 (Software development guide). Each version progresses through 8 DevOps phases (Plan, Code, Build, Test, Release, Deploy, Operate, Monitor), with teams alternating weekly focus. Planning is emphasized as reducing rework and debugging by 90%.131415

Weekly Status and Deliverables

Students submit three deliverables per week: (1) Weekly status (every week) documenting individual work, teamwork, planning, and 5+ stories worth 10 points each, (2) Role planning (even weeks) outlining role-specific responsibilities, and (3) Feature planning (odd weeks) detailing feature implementation. The weekly status takes approximately 1 hour initially, reducing to 15 minutes as students develop a pattern. All work is linked through Markdown cross-references in GitHub.161718

Grading Philosophy and Flexibility

The instructor uses a lenient grading approach for the first 2-3 weeks while students learn the system. The course is framed as a learning class, not a certification class, allowing students to revise and resubmit work to improve grades. The instructor distinguishes between reasons (legitimate obstacles) and excuses (well-planned lies), showing flexibility for genuine difficulties but maintaining accountability for commitment.
