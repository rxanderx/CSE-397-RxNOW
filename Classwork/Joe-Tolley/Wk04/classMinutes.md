5/11 Class Meeting Minutes

* DevOps Workflow and Two-Week Sprint Cycle

DevOps implements a continuous cycle breaking into even weeks (monitoring, planning, development, build) and odd weeks (test, release, deploy, operate). The methodology emphasizes maintainable code development, automated and manual testing, packaging for release with version descriptions, and continuous monitoring of logs and user feedback. Quality is maintained through the intersection of development, operations, and quality assurance, with schedule, budget, and resources as key factors.2

* Career Readiness and Job Search Strategy

Students must complete the 2-Hour Job Search framework including: creating a LAMP list (companies, advocacy contacts, motivation, posting frequency), conducting informational interviews using three core questions ('What do you do?', 'How can I do it?', 'Who else does it?'), and following a 10-day contact protocol (initial email, 3-day follow-up, 7-day final attempt). Students should complete 11 informational interviews, create 3 LinkedIn posts showcasing portfolio work, and apply to jobs to prepare for post-graduation employment.3

* Weekly Project Requirements and Documentation

Students must submit weekly status reports with links to evidence including: class meeting minutes (using GitHub templates), career readiness certificate updates, job search activities, role/feature planning, design documents, team meeting minutes, and product features/testing completed. The design document should be organized by front-end, back-end, database with interfaces, and installation/delivery instructions. White papers are required for research topics (architecture, lifecycle methodologies, networking, QA) comparing multiple approaches and recommending the best fit for the project.4

* Minimum Viable Product Features and Directory Structure

Every application should include common features: profile management (create, update, archive, read), session/login management, permission management (role-based access), user communication (notifications, comments, messaging), workflow management (status tracking), data management (single records and collections), reporting/analytics, and billing/ROI (subscriptions, shopping cart, transactions). The recommended directory structure includes source (front-end, back-end, database), docs, tests, data, project management, software engineering artifacts organized by issue number, and deliverables.5

* Git Workflow and Branch Management

Students must merge their work into weekly development branches (Week 3, Week 5, etc.) by specified deadlines. The instructor performs bi-weekly merges from individual branches into the development branch, then updates the current working branch. Branches should not be deleted as they provide evidence of work throughout the semester. GitHub Insights shows individual contribution metrics. The current working branch (Week 5) contains identical content to the development branch and is where all current work should be committed.
